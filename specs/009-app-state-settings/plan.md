# Implementation Plan: Application State & Settings

**Branch**: `009-app-state-settings` | **Date**: 2026-02-21 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/009-app-state-settings/spec.md`

## Summary

Implement VoxState as the central application state object implementing
GPUI's Global trait, enabling `cx.global::<VoxState>()` access from any
GPUI context. VoxState manages: user settings (23-field JSON-persisted
config with atomic write), SQLite transcript history (CRUD with search
and secure delete), application readiness tracking (Downloading →
Loading → Ready state machine), and pipeline state. All data stays
local — no network, no telemetry.

Key architectural decisions:
- gpui added as **required** dependency to vox_core (Principle XI)
- TranscriptStore extended with search/delete/clear_secure, shared via
  `Arc<TranscriptStore>` between VoxState and pipeline
- Data directory unified under `com.vox.app/` alongside existing models
- Settings atomic write (temp file + rename) prevents corruption

## Technical Context

**Language/Version**: Rust 2024 (1.85+)

0.12, rusqlite 0.38 (bundled), serde/serde_json 1, dirs 5, tokio 1.49,
uuid 1, tracing 0.1
**Storage**: SQLite via rusqlite 0.38 (bundled) + JSON settings file
**Testing**: `cargo test -p vox_core --features cuda` (Windows) /
`--features metal` (macOS)
**Target Platform**: Windows (CUDA) + macOS (Metal)
**Project Type**: Three-crate Rust workspace (vox, vox_core, vox_ui)
**Performance Goals**: Settings load/save < 10ms, transcript save < 5ms,
transcript search < 50ms (10k entries), VoxState init < 50ms
**Constraints**: All local processing, < 500MB RAM, zero network calls,
zero telemetry, zero cloud sync
**Scale/Scope**: Single user, up to 10,000 transcript entries, < 10MB
database
